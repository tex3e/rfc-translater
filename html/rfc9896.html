SVG in RFCs
RFC における SVG
Abstract
概要

This document defines policy for the inclusion of Scalable Vector Graphics (SVG) in the definitive versions of RFCs and relevant publication formats. It contains policy requirements from RFC 7996 but removes all requirements related to using a specific SVG profile or implementation code. It also makes the RFC Production Center (RPC) responsible for decisions about SVG tooling and implementation.

この文書は、RFC および関連する出版形式の決定版にスケーラブル ベクター グラフィックス (SVG) を含めるためのポリシーを定義します。これには RFC 7996 のポリシー要件が含まれていますが、特定の SVG プロファイルまたは実装コードの使用に関連するすべての要件が削除されています。また、RFC Production Center (RPC) が SVG ツールと実装に関する決定を担当するようになります。

This document obsoletes RFC 7996.

この文書は RFC 7996 を廃止します。

1. Introduction
1. はじめに

This document defines policy for the inclusion of Scalable Vector Graphics (SVG) in the definitive versions of RFCs and relevant publication formats defined in [RFC9720]. It contains policy requirements taken from [RFC7996] but removes all requirements related to using a specific SVG profile or implementation code.

この文書は、[RFC9720] で定義されている RFC および関連出版フォーマットの決定版に Scalable Vector Graphics (SVG) を含めるためのポリシーを定義します。これには [RFC7996] から取得したポリシー要件が含まれていますが、特定の SVG プロファイルまたは実装コードの使用に関連するすべての要件が削除されています。

SVG has been developed by the World Wide Web Consortium (W3C); see [SVG].

SVG は World Wide Web Consortium (W3C) によって開発されました。[SVG]を参照してください。

The RFC Production Center (RPC) is responsible for making decisions about SVG tooling and implementation. The RPC may use the content of [RFC7996] as a starting point for those decisions, but they are not bound by [RFC7996]. In addition, the RPC may change elements of the implementation as needed to support the RFC authoring community as long as those changes are aligned with the policy requirements in this document.

RFC プロダクション センター (RPC) は、SVG ツールと実装に関する決定を行う責任があります。RPC は、それらの決定の出発点として [RFC7996] の内容を使用する可能性がありますが、それらは [RFC7996] に拘束されません。さらに、RPC は、その変更がこの文書のポリシー要件に沿っている限り、RFC オーサリング コミュニティをサポートするために必要に応じて実装の要素を変更することがあります。

2. Policy Requirements
2. ポリシー要件

Decisions about SVG tooling and implementation are made by the RPC and must adhere to the policy requirements in this document:

SVG ツールと実装に関する決定は RPC によって行われ、このドキュメントのポリシー要件に従う必要があります。

* SVG drawings may be included in RFCs to help explain a concept more clearly, but they should not be the only representation of that concept. A good-faith effort should be made to ensure that descriptions of concepts -- which might include protocols, formats, or system architectures -- are fully represented in the text of the RFC. At minimum, SVG drawings should be consistent with the descriptions in the text of the RFC.

* 概念をより明確に説明するために SVG 図面が RFC に含まれる場合がありますが、それがその概念の唯一の表現であってはなりません。プロトコル、フォーマット、システム アーキテクチャなどの概念の説明が RFC の本文に完全に表現されていることを保証するために、誠意を持って努力する必要があります。少なくとも、SVG 図面は RFC 本文の説明と一致している必要があります。

* SVG drawings must not include animation or interactive features. SVG drawings should include only limited reactive design elements (scaling, dark/light mode, and perhaps minor adjustments to allow for variations in display technology). The intent of this is to ensure that the diagram's meaning is not altered.

* SVG 図面には、アニメーションやインタラクティブな機能を含めることはできません。SVG 図面には、限られたリアクティブ デザイン要素 (スケーリング、ダーク/ライト モード、および表示テクノロジの変化を考慮したわずかな調整) のみを含める必要があります。この目的は、図の意味が変更されないようにすることです。

* Images and diagrams in RFCs should be successfully rendered and understood by the widest audience possible. To that end, the RPC may prohibit the use of SVG features that are known to lack support on common devices, that do not render on small or low-resolution screens, or that could make diagrams less comprehensible for any significant readership. In particular:

* RFC 内の画像と図は、可能な限り幅広い読者に適切に表示され、理解される必要があります。そのために、RPC は、一般的なデバイスでサポートされていないことが知られている SVG 機能、小さな画面や低解像度の画面ではレンダリングされない、または重要な読者にとって図が理解しにくくなる可能性がある SVG 機能の使用を禁止する場合があります。特に：

- SVG drawings must not contain pointers to external resources.

- SVG 図面には、外部リソースへのポインタが含まれていてはなりません。

- SVG drawings must not contain executable script.

- SVG 図面には、実行可能なスクリプトを含めることはできません。

- SVG drawings should be as accessible as possible to people with visual disabilities, including those who have color blindness, those who need to scale or change fonts, and those who use screen-reading software. The RPC will refer to the W3C Accessibility Guidelines [WAI] when making decisions regarding accessibility.

- SVG 図面は、色覚異常を持つ人、フォントの拡大縮小や変更が必要な人、画面読み上げソフトウェアを使用する人など、視覚障害を持つ人にとって可能な限りアクセシブルである必要があります。RPC は、アクセシビリティに関する決定を行う際に、W3C アクセシビリティ ガイドライン [WAI] を参照します。

* Authors may include multiple versions of images or diagrams in RFCXML [RFC9720]. Publication formats should present the versions best suited to each format. In many cases, that will be an SVG.

* 作成者は、RFCXML [RFC9720] に複数のバージョンの画像または図を含めることができます。出版形式は、各形式に最適なバージョンを提示する必要があります。多くの場合、それは SVG になります。

* SVG vocabulary and implementation may change over time. Changes are not required to remain backwards compatible, although maintaining compatibility where possible is encouraged.

* SVG の語彙と実装は時間の経過とともに変更される可能性があります。下位互換性を維持するために変更を行う必要はありませんが、可能な限り互換性を維持することが推奨されます。

The RPC is authorized to place constraints on SVG usage in RFCs for both technical and editorial reasons in order to ensure that published RFCs meet the above policy and to provide consistency across the RFC Series. The RPC must document the acceptable usage of SVG, and all changes to decisions about SVG tooling and implementation must be widely communicated to the RFC author community using mailing lists or other means.

RPC は、公開された RFC が上記のポリシーを満たしていることを確認し、RFC シリーズ全体で一貫性を提供するために、技術的および編集上の理由から、RFC での SVG の使用に制約を設ける権限を与えられています。RPC は SVG の許容可能な使用法を文書化する必要があり、SVG ツールと実装に関する決定に対するすべての変更は、メーリング リストまたはその他の手段を使用して RFC 作成者コミュニティに広く通知する必要があります。

3. Implementation Guidance
3. 実施ガイダンス

The RPC is expected to solicit community input before making decisions and to publicly explain their reasoning.

RPC は決定を下す前にコミュニティからの意見を求め、その理由を公的に説明することが期待されています。

Documentation produced by the RPC should describe the technical and editorial constraints that apply to SVG and provide RFC authors with guidance on how to produce diagrams that meet those constraints.

RPC によって作成されるドキュメントは、SVG に適用される技術的および編集上の制約を説明し、それらの制約を満たす図を作成する方法に関するガイダンスを RFC 作成者に提供する必要があります。

The RPC's implementation should strive to allow SVG drawings produced by widely used drawing tools. Where possible, implementation decisions should focus on specifying what is disallowed rather than attempting to specify exactly what is allowed.

RPC の実装では、広く使用されている描画ツールで作成された SVG 描画を許可するよう努める必要があります。可能な場合、実装の決定は、何が許可されるかを正確に指定するのではなく、何が禁止されるかを指定することに重点を置く必要があります。

The RPC should periodically review and revise their practices.

RPC は定期的にその実践を見直し、修正する必要があります。
